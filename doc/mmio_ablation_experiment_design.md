# MMIO Ablation Experiment Design

## Objective

This document defines a reproducible ablation protocol for LCMHAL MMIO-related function discovery and replacement, focusing on:

- Contribution of feedback loop
- Contribution of static analysis
- Contribution of replacement strategy combinations

The goal is not full driver-directory coverage, but accurate identification of **MMIO-involved functions** that matter for build/emulation/task progress.

## Scope And Inputs

- Testcase root: `testcases/server`
- Per-demo stats source: `classify_stats_output.json`
- Batch summary source: generated by `python main.py classify-stats ...`
- MMIO metric script: `scripts/mmio_ablation_metrics.py`

Optional:

- Manual gold labels for sampled demos: `run/mmio_gold_labels.json`

## Experiment Matrix

### Core 2x2 Ablation (Recommended First)

Use this as minimum publishable setup:

1. No feedback + No static analysis
2. No feedback + With static analysis
3. With feedback + No static analysis
4. With feedback + With static analysis

Suggested repeat count per config: 3 to 5 runs.

### Replacement Strategy Ablation (Second Stage)

On top of the best setting from 2x2:

- Template-only replacement
- LLM replacement enabled
- Conservative replacement constraints on/off
- Historical replacement reuse on/off

Use factorized combinations and compare effectiveness vs risk.

## Reproducibility Controls

Keep the following fixed across all configs:

- Same testcase list
- Same timeout and retry budget
- Same max rounds
- Same model version
- Same parallelism and scheduling policy
- Same success criteria

## Metrics

All metrics below are emitted by `scripts/mmio_ablation_metrics.py` or can be derived from its JSON output.

### 1) Predicted Count

- Definition: number of functions predicted as MMIO-positive in one demo.
- Default positive rule: `function_type in {INIT, IRQ, RECV, LOOP}`.
- Meaning: absolute discovery volume of MMIO-related targets.

### 2) Predicted Ratio

- Definition: `predicted_count / total_functions`.
- Meaning: normalized MMIO discovery intensity; enables fair cross-demo comparison.
- Notes:
  - `predicted_ratio_micro`: aggregate ratio over all functions.
  - `predicted_ratio_macro`: average of per-demo ratios.

### 3) Has-Replacement-True Count

- Definition: count of functions with `has_replacement=true`.
- Meaning: how many discovered functions are considered replaceable by classifier output.
- Interpretation:
  - High value may indicate stronger intervention capability.
  - Needs to be read together with downstream success/failure to avoid over-replacement bias.

### 4) ReplacementUpdate Count

- Definition: number of functions with at least one `replacement_update_*` log (latest per function).
- Meaning: how many functions actually received replacement-update operations in pipeline execution.
- Interpretation:
  - Better proxy for real replacement actions than classifier intent alone.

### 5) Parse Error Count

- Definition: count of malformed/unreadable classification logs.
- Meaning: data quality and pipeline stability indicator.
- Requirement:
  - Must be near zero for trustworthy comparisons.

### 6) Gold Precision (Optional)

- Requires manual labels (`--gold-map`).
- Definition: `TP / predicted_count`.
- Meaning: proportion of detected MMIO targets that are truly MMIO-relevant.

### 7) Gold Recall (Optional)

- Requires manual labels.
- Definition: `TP / gold_count`.
- Meaning: ability to recover truly MMIO-relevant functions.

### 8) Gold F1 (Optional)

- Definition: harmonic mean of precision and recall.
- Meaning: balanced quality score for MMIO detection.

## Recommended Reporting Structure

### Table A: 2x2 Ablation Overall

Columns:

- Config
- predicted_ratio_micro
- has_replacement_true_count
- replacement_update_count
- parse_error_count
- task/emulate success rate (external runtime metric)

### Table B: Per-Platform Breakdown

Group demos by platform (`stm32`, `nxp`, `rtthread`, `zephyr`) and report:

- mean predicted ratio
- mean replacement update count
- success rate

### Table C: Gold-Label Evaluation (Sampled)

Columns:

- Config
- macro precision
- macro recall
- macro F1
- sample size

## Execution Commands

### Single-root metrics

```bash
python scripts/mmio_ablation_metrics.py \
  --root testcases/server \
  --output-prefix run/mmio_metrics_server
```

### Compare two ablation outputs

```bash
python scripts/mmio_ablation_metrics.py \
  --root-a <result_root_A> \
  --root-b <result_root_B> \
  --output-prefix run/mmio_ablation_compare
```

### With gold labels

```bash
python scripts/mmio_ablation_metrics.py \
  --root testcases/server \
  --gold-map run/mmio_gold_labels.json \
  --output-prefix run/mmio_metrics_with_gold
```

## Gold Label File Format

```json
{
  "zephyr/stm32/eth_dhcpv4_nucleo_f767zi": ["funcA", "funcB"],
  "nxp/NXP_I2C_BareMetal": ["I2C_MasterTransfer", "CLOCK_SetMux"]
}
```

Each key is demo relative path (under `--root`), value is list of MMIO-positive function names.

## Threats To Validity

- Proxy target bias: `function_type`-based positives are heuristic if no gold labels are used.
- Runtime confounders: network/API instability can affect replacement outcomes.
- Platform heterogeneity: different SDK/RTOS may naturally shift MMIO density.
- Data freshness: stale logs can mix results from different runs if not cleaned.

## Practical Recommendations

- Start from the 2x2 matrix and report micro+macro together.
- Keep `parse_error_count=0` as a hard acceptance gate.
- Add a small gold set per platform (even 30 to 50 functions each) to anchor claims.
- Pair detection metrics with runtime success metrics before drawing replacement conclusions.
